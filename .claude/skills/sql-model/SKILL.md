## When to use

For any entity that:
- lives in a SQL table,
- has fields **generated by the DB** (auto-incremented id, timestamps) that you don't provide on insert,
- has **sensitive** fields (password hash, tokens) that must be masked in HTTP output / logs,
- needs to be exposed over an HTTP API **and** manipulated by a repository, without duplicating the schema.

`Model.Class` generates **several views** of the same model from a single declaration:
- `Auth` — full type (DB read)
- `Auth.insert` — without the `Generated` fields
- `Auth.update` — partial over the non-generated fields
- `Auth.json` / `Auth.jsonCreate` / `Auth.jsonUpdate` — HTTP-side versions, **without the `Sensitive` fields**

## Procedure

1. Import `Model` from `@effect/sql`.
2. Declare the class: `class Xxx extends Model.Class<Xxx>("Xxx")({...}) {}`.
   - First argument = name (used in traces).
   - The trailing `{}` of the class is mandatory.
3. For each column, use **a branded type** (see branded-types) rather than a primitive. Consistency between domain ↔ persistence.
4. Wrap based on the column's role:
   - `Model.Generated(X)` — set by the DB. Present on read, absent on `insert`.
   - `Model.Sensitive(X)` — masked in the `json*` variants. Stays in the clear in the DB variants (otherwise the repo can't write it).
   - `Schema.NullOr(X)` — nullable column.
   - Composable: `Model.Generated(Schema.Date)` for a DB timestamp.
5. Expose over the HTTP API via `Auth.json` (read), `Auth.jsonCreate` (POST), `Auth.jsonUpdate` (PATCH).

## Pitfalls

- **`Model.Sensitive` only masks the `json*` variants**. The repo's `findById` still returns the field — that's intentional, otherwise it would be impossible to compare a hash. Don't rely on it for application-level security; it's just a safety net against HTTP / log leaks.
- **`Generated` fields on insert**. If you do `repo.insert(payload)` and `payload` is typed as `Auth` (not `Auth.insert`), the compiler lets you pass an `id` that the DB will silently ignore or reject. Always type inputs with the right variant.
- **Branded type ≠ DB constraint**. The brand is compile-time only; it doesn't generate the migration. The `email` column must have its own `CHECK` / `UNIQUE` on the DB side independently.
- **`Schema.NullOr` vs. missing column**. `NullOr` = the column exists and can be `NULL`. If the field can be *omitted* from the payload (partial insert), that's `Schema.optional`, not `NullOr`.
- **`Model.Class` name vs. table name**. The first argument is NOT the table name — it's just a label for traces. The table name is passed to the sql-repository via `Model.makeRepository`.

## Example

```ts
import { Model } from "@effect/sql"
import { Schema } from "effect"
import {
  AuthId,
  Email,
  PasswordHash,
  IsActive,
  EmailVerified,
  LastLogin,
  SubscriptionDate,
  SubscriptionStatusIdId
} from "../../domain/AuthType.js"

export class Auth extends Model.Class<Auth>("Auth")({
  id: Model.Generated(AuthId),
  email: Email,
  passwordHash: Model.Sensitive(PasswordHash),
  isActive: IsActive,
  emailVerified: EmailVerified,
  lastLogin: Schema.NullOr(LastLogin),
  subscriptionDate: Schema.NullOr(SubscriptionDate),
  subscriptionStatusId: SubscriptionStatusIdId,
  createdAt: Model.Generated(Schema.Date),
  updatedAt: Model.Generated(Schema.Date)
}) {}

// Variants generated automatically:
// Auth            → { id, email, passwordHash, isActive, ..., createdAt, updatedAt }
// Auth.insert     → { email, passwordHash, isActive, ... }              (without Generated)
// Auth.update     → Partial<{ email, passwordHash, ... }>
// Auth.json       → { id, email, isActive, ..., createdAt }             (without Sensitive)
// Auth.jsonCreate → { email, isActive, ... }                            (without Generated nor Sensitive)
// Auth.jsonUpdate → Partial<{ email, isActive, ... }>
```
